# Risks And Decisions

## 风险 1：Codex 日志格式变化

影响：

- MVP 依赖本地 JSONL 日志。
- 如果 Codex 改字段，统计会失效。

缓解：

- 日志解析逻辑集中在一个模块。
- 验收报告记录当前已验证字段。
- 后续 SQLite ingestion 保留原始 event hash，方便重建。

## 风险 2：模型名归因不准

影响：

- `token_count` 事件本身未必总带 model。
- MVP 需要从同一会话的 `turn_context.model` 推断。

缓解：

- dashboard 标注模型归因为会话上下文推断。
- gateway 阶段直接从请求 model 字段记录，解决根因。

## 风险 3：使用时长定义不清

影响：

- 用户想看的“使用时间”可能是会话时间、模型响应耗时、流式输出时间。

决策：

- MVP 使用 session active time，定义为第一次 token 事件到最后一次 token 事件。
- Gateway 阶段新增 request latency、first token latency、stream duration。

## 风险 4：Claude Code 和 Cursor 不一定支持本地网关

影响：

- 如果工具不支持自定义 base URL，就无法直接走本地 gateway。

缓解：

- 阶段 0 单独调研每个工具。
- 每个工具保留日志型、网关型、系统代理型三种候选路径。
- 不在未验证前承诺具体接入方式。

## 风险 5：隐私和安全

影响：

- 本地网关理论上能看到 prompt、代码、响应。

决策：

- 默认只记录 usage metadata。
- 默认不记录 prompt 和 response 正文。
- dashboard 默认只绑定 `127.0.0.1`。
- API key 本地存储，后续再做加密。

## 风险 6：费用计算不准确

影响：

- provider 价格经常变化。
- cached input、reasoning token、batch、不同区域价格可能不同。

决策：

- MVP 不做费用。
- Gateway 阶段先支持手动价格表。
- 费用字段标记为 estimated。

## 风险 7：重复计数

影响：

- Codex 日志可能被多次扫描。
- 同一 token_count 事件可能重复读入。
- 实测发现同一 session 内会出现相同 `total_token_usage` 快照；直接累加每条 `last_token_usage` 会明显重复计数。

缓解：

- MVP 实时扫描时按 session 和 timestamp 排序读取 `token_count`。
- 使用 `payload.info.total_token_usage` 的相邻累计快照 delta 作为主统计口径。
- 相同累计快照视为重复快照，跳过。
- `last_token_usage` 只作为辅助校验，不直接累加。
- SQLite 阶段使用 event id/hash 做唯一约束。
